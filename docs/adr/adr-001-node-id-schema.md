# ADR 001: 8-Character Alphanumeric Cluster Node ID Schema

## Status

Accepted

## Context

Each node in the AI Relay cluster needs a stable, human-readable identifier used in
logs, the dashboard, API paths, and persisted state. The identifier must be short
enough to display comfortably in URLs and CLI output, yet have enough entropy to
avoid collisions at expected cluster sizes. The cluster assigns the ID; nodes do not
choose their own.

## Decision

Adopt an **8-character alphanumeric node ID** generated by the control plane when a
node is first approved into the cluster.

### Exact format

- **Length:** exactly 8 characters (fixed width).
- **Alphabet:** 27 symbols:
  - Uppercase letters `A–Z` excluding the ambiguous letters `I`, `L`, `O`, `S`, `Z`
  - Digits `0–9` excluding the ambiguous digits `0`, `1`, `2`, `5`
  - Explicit alphabet: `ABCDEFGHJKMNPQRTUVWXY346789`
- **Case handling:** IDs are generated in uppercase. User input is compared
  case-insensitively and then normalized to uppercase for storage and display.
- **Excluded ambiguous characters:**
  - `0` (zero) vs. `O` (letter O)
  - `1` (one) vs. `I` (letter I) and `L` (letter L)
  - `2` (two) vs. `Z` (letter Z)
  - `5` (five) vs. `S` (letter S)

### Regex

Canonical validation regex for APIs, config files, and CLI input:

```regex
^[ABCDEFGHJKMNPQRTUVWXY346789]{8}$
```

For case-insensitive input validation, normalize to uppercase first and then apply
the canonical regex.

### Generation

```python
import secrets

NODE_ID_ALPHABET = "ABCDEFGHJKMNPQRTUVWXY346789"
NODE_ID_LENGTH = 8
NODE_ID_MAX_RETRIES = 100


def generate_node_id() -> str:
    return "".join(secrets.choice(NODE_ID_ALPHABET) for _ in range(NODE_ID_LENGTH))
```

### Collision-avoidance strategy

1. IDs are generated centrally by the control plane, never by nodes themselves.
2. Before assignment, the control plane queries the node registry for an existing
   ID. If a collision occurs, a new candidate is generated (rejection sampling).
3. The registry enforces a unique constraint on `node_id` so concurrent assignment
   attempts for the same ID are rejected at persistence time.
4. Assigned IDs are persisted atomically before the node is informed of its ID,
   preventing double allocation.
5. If no unused ID can be allocated after `NODE_ID_MAX_RETRIES` attempts, the
   operation fails with an explicit error.

### Uniqueness guarantee

- A `UNIQUE` index on `nodes.node_id` guarantees that two approved nodes can never
  share the same ID in the registry.
- Centralized generation plus rejection sampling makes the practical probability of
  assignment failure effectively zero for any cluster size that fits in the registry.

### Keyspace

With 27 symbols over 8 positions:

```
27^8 = 282,429,536,481 ≈ 2.82 × 10^11
```

For 100,000 nodes, the birthday-problem collision probability before rejection
sampling is approximately:

```
P ≈ n^2 / (2 × N) = (10^5)^2 / (2 × 2.82 × 10^11) ≈ 1.8 × 10^-2
```

Rejection sampling reduces the practical collision probability to zero.

## Lifecycle moment

A node receives its ID at **first approval**, not at first network contact.

1. Node boots and connects to the control plane with a temporary registration
   secret.
2. An operator or automated policy approves the node via the dashboard or API.
3. On approval, the control plane allocates an unused 8-character ID.
4. The ID is returned in the approval response and becomes the node's persistent
   identity.
5. The node stores the ID locally (see Persistence) and includes it in subsequent
   heartbeats and API calls.

## Persistence

- **Database / registry primary key:** `node_id` in the `nodes` table.
- **Registry key pattern:** `nodes/{node_id}` (e.g. etcd key, object path, or file
  path segment).
- **Local node file:** `~/.relay/node_id` (or `/var/lib/relay/node_id` for systemd
  deployments), written after first approval.
- **API path parameter:** `/api/nodes/{node_id}`.

## Examples

Valid IDs:

```
A7B3C9D4
HJKM6P8R
TYYX3478
```

Invalid IDs:

```
a7b3c9d4      # lowercase (normalize to uppercase or reject)
A7B3C9D       # 7 characters
A7B3C9D4E     # 9 characters
A7B0C9D4      # contains forbidden 0
A7BIC9D4      # contains forbidden I
A7B5C9D4      # contains forbidden 5
A7BZC9D4      # contains forbidden Z
```

## Consequences

- URLs, log lines, and CLI commands stay short and readable.
- Excluding ambiguous characters reduces support burden and transcription errors.
- Fixed width simplifies formatting, database indexing, and UI alignment.
- Centralized allocation guarantees uniqueness without coordination overhead.
- The 27-symbol alphabet still provides ~2.8 × 10^11 possible IDs, sufficient for
  very large clusters.

## Open questions

1. Should the local node file be plaintext or a small JSON blob containing
   `{node_id, cluster_id, approved_at}`?
2. Do we need a "node ID reservation" API for pre-provisioning hardware?
3. Should revoked/deregistered node IDs ever be recycled? (Recommended: no, to keep
   audit trails clean.)
